# Codex-DR Sandbox Architect Handoff

Status: active local translation
Date: 2026-04-22

## Why This Exists

The Principal clarified that the immediate sandbox lane is a full attempt at
Grep-system-and-performance parity using Codex/terminal-agent machinery. The
target includes adequacy backpressure, recursive research, benchmark pressure,
visible tools, event-sourced custody, compaction provenance, and proof runs.

The pharma service runtime remains a later lane. The sandbox must inform it,
while first proving the research engine against benchmark-facing standards.

## Decision

Create the first architect workspace at `sandbox/codex-dr/`.

This keeps the architect close to the root charter, root plan, authority ledger,
Grep references, legacy proof receipts, and root development commands. The
workspace gets its own `AGENTS.md` and `README.md` so future agents can enter
through the sandbox target without being steered by Program 90 or old NLSpecs.

Hindsight correction: the workspace also needs an explicit bootstrap doctrine.
Repo memory alone does not make the architect safe to run. The architect needs
a provider-off first boot, token-burn firewall, root-gravity firewall, and
skill gates before model-backed execution.

A separate repository or worktree becomes appropriate after one of these
conditions appears:

- substantial generated run volume
- private benchmark data
- customer material
- secret-bearing configs
- parallel implementation teams with conflicting write scopes
- evidence that the root repo is confusing future agents

## Root AGENTS Interaction

The root `AGENTS.md` remains the global map. It gives repo hygiene, authority
discipline, and commands.

The sandbox `AGENTS.md` narrows the mission for agents working under
`sandbox/codex-dr/`. Future architect launches should use that folder as cwd
where possible, so the local instructions become the immediate handoff surface.

The root map must point to the sandbox for this lane. Otherwise a future agent
may read historical Program 90 surfaces and inherit the wrong centre.

## Architect Memory Contract

The architect must remember through repo-local artifacts:

- `sandbox/codex-dr/AGENTS.md`
- `sandbox/codex-dr/README.md`
- `sandbox/codex-dr/docs/BOOTSTRAP_DOCTRINE.md`
- `sandbox/codex-dr/docs/ARCHITECT_HANDOFF.md`
- `docs/exec-plans/active/codex_dr_sandbox_architect_handoff.md`
- this design memo
- the bead entries created for this lane

Chat is not an authority surface.

## Required First Move

Complete the Bootstrap Autonomy Gate in
`sandbox/codex-dr/docs/BOOTSTRAP_DOCTRINE.md`.

The gate requires:

- centre-lock receipt
- root-gravity firewall
- provider-off run-bundle skeleton
- fake planner, branch, reviewer, re-entry, report, compaction, claim ledger,
  benchmark-score placeholder, and allowed-claims flow
- validators that fail on missing custody or missing re-entry
- token manifest template for later provider-backed runs

After that, perform a Benchmark Acquisition Audit.

The audit must cover:

- DRACO
- DeepSearchQA
- DeepResearch Bench
- Parcha-published benchmark material where available
- source URLs and versions
- licenses and use restrictions
- dataset sizes and case-selection rules
- evaluator shape and scoring rubrics
- local reproduction requirements
- gaps between public benchmark material and any Grep/Parcha private setup
- claims permitted after acquisition
- claims blocked until external evidence exists

The audit calibrates the target. It does not authorize benchmark execution by
itself.

## Full Sandbox Target

The sandbox architect owns the full system attempt:

- planner ratification of request and telos
- planner research before Plan File emission
- task graph creation
- branch spawning and scoped terminal-agent execution
- branch/subagent evidence return
- recursive deepening against adequacy criteria
- synthesis and contradiction reconciliation
- reviewer and fact-checker file outputs
- reviewer-driven research re-entry
- single-voice report generation
- benchmark scoring
- event log and content-addressed artefact custody
- compaction receipts
- claim ledger
- allowed-claims output

Smaller slices may reduce risk during implementation. They do not redefine the
finish line.

The first slice must be local and provider-off. Paid or model-backed runs begin
only after the bootstrap validator and token manifest rules exist.

## Service Runtime Bridge

The sandbox is not the product runtime. It is the proving lane for the research
engine.

The later service runtime must still specify API boundaries, tenant separation,
data policy, job queues, durable workers, provider billing, source connector
policy, secrets, audit export, and client-facing report surfaces.

The bridge artifact at the end of the sandbox lane should classify each
contract as:

- promotable as written
- promotable after hardening
- sandbox-only
- replaced by service-runtime design

## Non-Claims

This memo does not claim Grep parity.

This memo does not authorize product deployment.

This memo does not ratify the root charter or root plan.

This memo gives future agents a durable entry surface for the sandbox lane.
