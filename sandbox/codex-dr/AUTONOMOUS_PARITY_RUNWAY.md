Status: active instruction surface

## Purpose

This document converts the Principal's current authorization into a durable
work runway for the Codex-DR sandbox.

The immediate objective is to build and improve a Codex CLI native deep-research
mesh until it can make evidence-backed DeepResearch Bench claims against the
current Grep target. The work is authorized to include design, implementation,
tests, live Codex role runs, benchmark source refresh, prompt and skill
iteration, file-based context-economy changes, evaluation, and improvement
loops, subject to the run-control and claim gates in this workspace.

## Completion Object

The object that must change is the Codex-DR sandbox. It should become a
self-driving engineering lane that can:

- run benchmark-shaped deep-research cases through a Grep-style topology;
- spawn scoped Codex role boxes under the live mesh harness;
- record file-return, transcript, event, context, adequacy, and claim custody;
- score outputs with the correct current benchmark evaluator path;
- convert scored failures into prompt, skill, scheduler, file-economy, or
  evaluator changes;
- rerun regression gates before promoting those changes;
- stop widening claims until proof artifacts permit the next claim.

## Assumptions

- Codex CLI is the intended agent substrate for the sandbox validation lane.
- The system target is Grep-style deep-research cognition, expressed as
  planner grounding, task graphing, branch research, file returns,
  pointer-first orchestration, adequacy recursion, reviewer re-entry, writing,
  scoring, and improvement.
- DeepResearch Bench is the main public benchmark lane currently under
  discussion.
- Benchmark truth is moving. The official DeepResearch Bench repository was
  updated on 2026-04-24, and its evaluator policy is changing before June 2026.
- A benchmark score only means something when the case set, evaluator, judge
  model, raw generated reports, retry policy, scorer outputs, and claim review
  are recorded.
- Grep parity means at least matching the current official Grep-v5
  DeepResearch Bench score on the same evaluator lane, with topology proof
  preserved in the run bundle.

If any of these assumptions fail, the next bead must surface the failure and
repair this runway before fanning out implementation.

## Teleological Ground

The sandbox is for building an operating deep-research mesh, then improving it
under benchmark pressure until its claims are earned.

Failure would include:

- building local harness machinery while benchmark scoring remains absent;
- running Codex agents without the Grep topology;
- producing reports without reviewer re-entry and adequacy pressure;
- treating blocked placeholder scores as benchmark evidence;
- improving prompts or skills without replay fixtures and regression checks;
- using stale leaderboard or evaluator facts to define parity;
- leaving future agents with conceptual docs and no ready bead ladder.

## Operating Loop

The engineering loop is:

1. Recover memory and read this runway.
2. Select the next ready bead.
3. State the parity rung being advanced.
4. Name the fake success condition.
5. Implement the smallest code, prompt, skill, manifest, or validator change
   that advances the rung.
6. Run targeted tests.
7. Run the relevant mesh or benchmark command when authorized by receipt.
8. Write proof artifacts.
9. Update allowed claims.
10. Create the next bead if the queue is empty and the next proof gap is clear.
11. Capture a short Open Brain memory when the run changes the runway.

The deep-research run loop is:

```text
INTENT
  -> PLAN
  -> TASK_GRAPH
  -> SPAWN_BRANCHES
  -> POINTER_READ
  -> SYNTHESIZE
  -> ADEQUACY_CHECK
  -> REENTER_OR_REVIEW
  -> WRITE_REPORT
  -> SCORE
  -> IMPROVE
  -> CLAIM_REVIEW
```

The improvement loop is:

```text
SCORE_OR_REVIEW_FAILURE
  -> FAILURE_TAXONOMY
  -> PATCH_CANDIDATE
  -> REPLAY_FIXTURE
  -> REGRESSION_GATE
  -> PROMOTION_OR_REJECTION
  -> RERUN
```
