# Bohr Trajectory Forensics

Status: forensic memo
Date: 2026-04-22
Scope: Codex-DR / Bohr DR mesh build

## Root Cause

Bohr did not initially live up to the Principal's teleological intention because
the actionable center of gravity was under-specified.

The intended target was the DR mesh: a Grep-style recursive deep-research
agent system validated through Codex CLI boxes. The durable handoff surfaces
named Grep parity, benchmark pressure, provider-off bootstrap, token/run
control, and custody, but they did not initially force every next action to
reproduce the three Grep topology figures as executable behavior.

That left Bohr with several plausible lower-order centers:

- provider-off fixtures;
- token and launch-control gates;
- benchmark acquisition and score blockers;
- service-runtime boundary classification;
- generic custody and claim validation;
- one-off smoke-run receipts.

All of those are useful support structures. None is the DR mesh.

That before/after is the important signal. The same agent began doing the

## Contributing Causes

### 1. Abstract Noun Drift

Terms like "Grep parity", "sandbox", "harness", and "benchmark lane" were too
compressible. Bohr could satisfy those terms by improving support surfaces
without building the recursive agent topology.

Leverage: name the object as DR mesh and define it by required behaviors, not
by inspiration source.

### 2. Support Surfaces Became The Work

Custody, launch control, token control, benchmark acquisition, and claim
boundaries became more explicit than the mesh runtime. The agent followed the
clearest interfaces.

Leverage: every support surface must say which mesh behavior it enables.

### 3. Visual Architecture Was Not Durable Authority

The Grep diagrams were known in conversation, but they were not initially
repo-local, parsed, and binding. Bohr received a textual mission while the
actual topology lived outside the agent's durable build contract.

Leverage: turn topology diagrams into a build charter immediately, including
figure-by-figure implications and invalid success conditions.

### 4. No Invalid-Success Gate Early Enough

The early handoff said what to build, but it did not aggressively block common
fake completions: docs-only closure, provider-off fixture closure, launch
control closure, service-runtime memo closure, and benchmark-discussion
closure.

Leverage: write invalid success conditions before beads or implementation.

### 5. First Executable Center Was Ambiguous

The first lane oscillated among benchmark acquisition, provider-off bootstrap,
boxed smoke, token controls, and scorer bridge. That made "progress" too easy
to redefine.

Leverage: define the first executable center as a command-level mesh sequence:
plan, branch, evaluate, review, re-enter, write, score, validate.

### 6. The Agent Lacked A Trajectory Auditor

There was no standing check that compared Bohr's last N artifacts against the
teleological target and asked whether the work was strengthening the right
center.

Leverage: run agent trajectory forensics when an autonomous builder starts
closing support work without producing target behavior.

## Reusable Leverage Moves

### Move 1: Noun Lock

Name the build object in the Principal's language. Here, the noun is DR mesh.
Every plan, bead, command, and acceptance test must use that noun until the
center is stable.

### Move 2: Diagram-Backed Build Charter

When topology diagrams are load-bearing, convert them into a repo-local charter
with:

- figure-by-figure parse;
- required behaviors;
- state machine implications;
- file/interface implications;
- invalid success conditions;
- immediate build center.

### Move 3: Support-Surface Demotion

Classify every support artifact as "serves the mesh" or "not current work".
Launch control, custody, benchmarks, and service-runtime memos cannot become
completion unless they exercise the mesh.

### Move 4: Invalid-Success Gate

Before implementation, write a short list of results that look like progress
and still miss the point. Use the list as a validator for plans, beads, and
agent updates.

### Move 5: Command-Level Center

Force the target into visible commands. For this lane:

- `mesh-init-case`
- `mesh-plan`
- `mesh-branch`
- `mesh-evaluate`
- `mesh-review`
- `mesh-reentry`
- `mesh-report`
- `mesh-score`
- `validate`

If these commands do not exist, the mesh is not yet executable.

### Move 6: Before/After Evidence Check

Do not infer that a correction worked. Inspect the next artifacts. In this case,
the correction worked when Bohr began adding mesh role adapters, pointer-first
receipts, staged CLI commands, and topology validators.
