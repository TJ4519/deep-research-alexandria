# PLAN_TO_CREATE_ALEXANDRIA

Canonical name: Alexandria Deep Research Runtime
Product layer: pharma competitive-intelligence researcher
Status: Architecture charter + execution plan seed, pending Principal ratification
Date: 2026-04-21

## CAVEAT: Ambition-First Operating Rule

This plan is intentionally ambitious. Alexandria is being designed as a
category-defining deep-research agent system, with Grep-class recursive research
behavior as the first checkpoint and stronger custody, replay, reconciliation,
and domain intelligence as the exceeding move.

Out-of-charter behavior:

- Reducing Alexandria to a filesystem context-economy project.
- Treating Grep parity as the destination.
- Treating old numbered NLSpecs as automatically authoritative after they have
  produced drift.
- Replacing recursive research cognition with report-generation polish.
- Using audit logs as a substitute for real evidence retrieval, contradiction
  handling, synthesis, and reviewer-driven re-entry.
- Scope trimming framed as practicality when it weakens the deep-research
  engine.

In-charter behavior:

- Preserve ambition while increasing proof discipline.
- Treat Grep as a donor engine for recursive deep-research cognition.
- Convert every bold capability claim into artifacts, checks, and reviewable
  receipts.
- Build the filesystem, event log, and content-addressed store as the native
  substrate for deep-research cognition.
- Treat pharma competitive intelligence as the first commercial wedge and the
  first domain where chain-of-custody must be product-grade.

Directive alignment statement:

"Alexandria exists to reach and exceed Grep-class recursive deep-research
coordination. Filesystem working memory, append-only event logging, and
immutable artefacts are the native substrate and custody machinery for that
system. They are not the whole telos."

## 1. Background And Origin

Alexandria began as a cleanroom backend runtime with numbered NLSpecs, proof
harnesses, Program 80-90 execution history, generated receipts, and a strong
runtime-authority instinct.

The current reimagining began after studying Grep's public deep-research writeup
and receiving third-party Charter, Commissioning Brief, and Context Pack
materials. Those materials exposed a center-of-gravity risk: Alexandria could
preserve runtime authority and file-backed context while missing the recursive
research engine that makes Grep powerful.

The plan exists to install the corrected center.

## 2. Core Thesis

Alexandria will be a recursive deep-research runtime for high-stakes
intelligence work.

Its first required checkpoint is Grep-class research cognition:

- planner-shaped decomposition
- tool-mediated context gathering before planning
- task graphs that separate parallel and sequential reasoning
- orchestrator-managed research recursion
- file-returning sub-agents
- markdown-mediated working memory
- recursive evidence retrieval
- synthesis and contradiction reconciliation
- reviewer pressure that can send the system back into research
- single-voice report construction

Its Alexandria-native extension is custody and replay:

- every orchestrator I/O hop is evented
- every control-flow decision is emitted through a mandatory decision tool
- every plan, task graph, analysis, evidence file, review, section, and report
  is an immutable content-addressed artefact
- every completed session is reconstructible from the event log and artefact
  store alone
- every important report claim can be traced back through sources, intermediate
  analysis, review pressure, and admission decisions

## 3. Strategic Objective

Build the first practical deep-research runtime whose cognition is recursive,
domain-shaped, reconstructible, and commercially defensible.

The initial product surface is pharma competitive intelligence. Alexandria
competes with the analyst layer by producing high-quality research reports with
claim-to-source chain-of-custody. It does not compete as a proprietary pharma
database.

Category-defining floor:

- Grep-class inner loop over planning, tasking, spawning, evidence return,
  adequacy assessment, and recursive deepening.
- Grep-class outer loop over report writing, review, fact-checking, and
  research re-entry.
- Claim-level custody for material report claims.
- First-class contradiction objects with sources, positions, adjudication state,
  unresolved status, and report treatment.
- Reviewer findings that compile into new research tasks rather than prose-only
  edits.
- Domain-shaped pharma CI decomposition and adequacy criteria.
- Session replay from event log and content-addressed artefacts without
  provider memory.
- Proof packs that show retrieval quality, claim coverage, contradiction
  handling, reviewer impact, and report improvement across loops.

If outcomes are parity-only, context-economy-only, or report-polish-only, the
program is off-charter.

## 3.1 Category-Creation Doctrine

Alexandria is pursuing a new class of deep-research system.

Required doctrine:

- Treat recursive evidence work as the engine.
- Treat filesystem working memory as the operating medium for long-horizon
  research cognition.
- Treat append-only event logging as the causal trace of orchestration.
- Treat content-addressed artefacts as custody objects.
- Treat contradiction reconciliation as core research work.
- Treat reviewer pressure as a research-control input.
- Treat pharma CI as the first domain where the system must prove commercial
  seriousness.
- Treat every capability claim as incomplete until it has a durable proof
  artifact.

Category-creation test:

- If a capability can be reproduced by a prompt-only report generator, it is
  insufficient.
- If a capability cannot survive session reconstruction, it is insufficient.
- If reviewer criticism cannot alter the research state, it is insufficient.
- If contradiction handling disappears into smooth prose, it is insufficient.
- If a pharma CI report cannot show claim-to-source custody, it is insufficient.

## 3.2 Impossible-By-Default Capability Index

The program must deliver capabilities that are not available by default in
ordinary agent stacks:

- Event-backed claim-to-source chain-of-custody.
- Reconstructible research sessions from files and event log alone.
- Mandatory decision tool calls for spawn, terminate, re-plan, deepen, and ship
  decisions.
- Pointer / analysis / evidence return contracts for sub-agent work.
- Review findings that become structured research tasks.
- Contradiction objects with explicit adjudication and report treatment.
- Evidence-thin claim detection that triggers recursive retrieval.
- Domain-shaped adequacy criteria generated at planning time.
- Single-voice synthesis over multi-branch research state.
- Provider-role separation across drafting, review, synthesis, and adversarial
  pressure.
- Proof packs for loop improvement, claim coverage, source admission, and
  reviewer impact.

## 4. Non-Negotiable Constraints

- Alexandria runs on the Anthropic Messages API with a custom orchestrator
  loop.
- Alexandria does not use the Claude Agent SDK agent loop.
- Alexandria does not use LangGraph as its state-machine runtime.
- The model client remains provider-agnostic so Claude and GPT can serve
  different cognitive roles.
- The filesystem is working memory during research.
- The event log is append-only and complete for orchestrator I/O hops.
- Artefacts are immutable and content-addressed.
- Runtime/provider hidden state is never required to reconstruct a completed
  session.
- The first commercial wedge is pharma competitive intelligence.
- The system is not a database product.
- The system is not a general developer framework.
- Cross-session recursive improvement is preserved optionality, not a v1 product
  promise.
