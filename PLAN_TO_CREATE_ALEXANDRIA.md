## CAVEAT: Ambition-First Operating Rule

This plan is intentionally ambitious. Alexandria is being designed as a
category-defining deep-research runtime, with Grep-class recursive research
coordination as the first checkpoint and stronger custody, replay,
reconciliation, and domain intelligence as the exceeding move.

Out-of-charter behavior:

- Reducing Alexandria to filesystem-based context economy.
- Treating Grep parity as the final destination.
- Treating old numbered NLSpecs as automatically authoritative after they have
  produced drift.
- Replacing recursive research cognition with report-generation polish.
- Using audit logs as substitutes for evidence retrieval, contradiction
  handling, synthesis, and reviewer-driven re-entry.
- Converting planning into beads before the governing point is stable.
- Scope trimming framed as practicality when it weakens the deep-research
  engine.

In-charter behavior:

- Preserve ambition while increasing proof discipline.
- Treat Grep as a donor engine for recursive deep-research cognition.
- Convert capability claims into artifacts, checks, and reviewable receipts.
- Build filesystem working memory, event logging, and content addressing as
  native machinery for research cognition.
- Treat pharma competitive intelligence as the first commercial wedge and the
  first domain where claim custody must be product-grade.
- Carry the corrected telos into charter clauses, authority rules, design docs,
  beads, and proof gates before implementation expands.

Directive alignment statement:

> Alexandria exists to reach and exceed Grep-class recursive deep-research
> coordination. Filesystem working memory, append-only event logging, and
> immutable content-addressed artefacts are the native substrate and custody
> machinery for that system. They are not the whole telos.

## 1. Background And Origin

Alexandria began as a cleanroom backend runtime with numbered NLSpecs, proof
harnesses, Program 80-90 execution history, generated receipts, and a strong
runtime-authority instinct.

The current reimagining began after studying Grep's public deep-research
writeups and receiving third-party Charter, Commissioning Brief, and Context
Pack materials. Those materials exposed a center-of-gravity risk: Alexandria
could preserve runtime authority and file-backed context while missing the
recursive research engine that makes Grep powerful.

The later Claude-in-a-Box article sharpened the substrate question. Grep's
current implementation evidence points to isolated Claude Agent SDK boxes with
selective skill loading, scoped MCP tools, session fork/resume, queued worker
execution, and a researcher-to-reporter handoff. Alexandria should learn from
that substrate while keeping its own authority layer.

The user then supplied the FrankenEngine plan as a donor artifact for plan
craft. The lesson was structural. A serious plan binds ambition, names
forbidden reductions, defines proof floors, separates donor extraction from
implementation, installs a method stack, and converts doctrine into phases,
risks, success criteria, and evidence contracts.

This plan installs the corrected center for Alexandria.

## 2. Core Thesis

Alexandria will be a recursive deep-research runtime for high-stakes
intelligence work.

Its first required checkpoint is Grep-class research cognition:

- planner-shaped decomposition
- tool-mediated context gathering before planning
- task graphs that separate parallel and sequential reasoning
- orchestrator-managed research recursion
- file-returning sub-agents
- markdown-mediated working memory
- recursive evidence retrieval
- synthesis and contradiction reconciliation
- reviewer pressure that can send the system back into research
- single-voice report construction

Its Alexandria-native extension is custody and replay:

- every orchestrator I/O hop is evented
- every control-flow decision is emitted through a mandatory decision tool
- every plan, task graph, analysis, evidence file, review, section, and report
  is an immutable content-addressed artefact
- every completed session is reconstructible from the event log and artefact
  store alone
- every material report claim can be traced through sources, intermediate
  analysis, review pressure, and admission decisions

## 3. Strategic Objective

Build the first practical deep-research runtime whose cognition is recursive,
domain-shaped, reconstructible, and commercially defensible.

The initial product surface is pharma competitive intelligence. Alexandria
competes with the analyst layer by producing high-quality research reports with
claim-to-source chain-of-custody. It does not compete as a proprietary pharma
database.

Category-defining floor:

- Grep-class inner loop over planning, tasking, spawning, evidence return,
  adequacy assessment, and recursive deepening.
- Grep-class outer loop over report writing, review, fact-checking, and
  research re-entry.
- Claim-level custody for material report claims.
- First-class contradiction objects with sources, positions, adjudication
  state, unresolved status, and report treatment.
- Reviewer findings that compile into new research tasks.
- Domain-shaped pharma CI decomposition and adequacy criteria.
- Session replay from event log and content-addressed artefacts without
  provider memory.
- Proof packs that show retrieval quality, claim coverage, contradiction
  handling, reviewer impact, and report improvement across loops.
- Branch execution that can use Claude Agent SDK boxes when isolated
  filesystem work, selected skills, MCP tool scopes, and fork/resume semantics
  are the best route to Grep-parity behavior.

If outcomes are parity-only, context-economy-only, or report-polish-only, the
program is off-charter.

## 3.1 Category-Creation Doctrine

Alexandria is pursuing a new class of deep-research system.

Required doctrine:

- Treat recursive evidence work as the engine.
- Treat filesystem working memory as the operating medium for long-horizon
  research cognition.
- Treat append-only event logging as the causal trace of orchestration.
- Treat content-addressed artefacts as custody objects.
- Treat contradiction reconciliation as core research work.
- Treat reviewer pressure as a research-control input.
- Treat pharma CI as the first domain where the system must prove commercial
  seriousness.
- Treat every capability claim as incomplete until it has a durable proof
  artifact.

Category-creation test:

- If a capability can be reproduced by a prompt-only report generator, it is
  insufficient.
- If a capability cannot survive session reconstruction, it is insufficient.
- If reviewer criticism cannot alter the research state, it is insufficient.
- If contradiction handling disappears into smooth prose, it is insufficient.
- If a pharma CI report cannot show claim-to-source custody, it is insufficient.

## 3.2 Missing-The-Point Doctrine

The planning discipline begins with synthesis as a comprehension test.

A model may summarize Alexandria fluently while carrying the wrong operating
model. The governing question is whether a future agent could build from that
synthesis and preserve the intended system.

Known miss classes:

- `SUBSTRATE_AS_TELOS`: filesystem, event log, or CAS is mistaken for the
  system's purpose.
- `TOPOLOGY_AS_TELOS`: visible parallel/sequential decomposition is mistaken for
  the recursive research cognition that produces and governs it.
- `EXEMPLAR_AS_STYLE`: a donor plan is copied rhetorically instead of studied
  for structural moves.
- `AUTHORITY_INERTIA`: old specs or proof receipts govern because they already
  exist.
- `TASK_EAGERNESS`: work becomes beads before the governing point is stable.
- `INTERFACE_INVENTION`: builders need to invent schemas, semantics, or control
  contracts during implementation.
- `DOMAIN_THINNESS`: pharma CI appears as a report topic while evidence
  standards, claim custody, and adequacy criteria remain generic.

Repair rule:

Repair the governing surface before execution. A miss is repaired by amending
the charter, plan, authority ledger, design document, proof gate, or bead
packet. A miss is not repaired by adding warnings to chat.

## 3.3 Impossible-By-Default Capability Index

The program must deliver capabilities that ordinary agent stacks do not provide
by default:

- Event-backed claim-to-source chain-of-custody.
- Reconstructible research sessions from files and event log alone.
- Mandatory decision tool calls for spawn, terminate, re-plan, deepen, and ship

- Pointer / analysis / evidence return contracts for sub-agent work.
- Review findings that become structured research tasks.
- Contradiction objects with explicit adjudication and report treatment.
- Evidence-thin claim detection that triggers recursive retrieval.
- Domain-shaped adequacy criteria generated at planning time.
- Single-voice synthesis over multi-branch research state.
- Provider-role separation across drafting, review, synthesis, and adversarial
  pressure.
- Proof packs for loop improvement, claim coverage, source admission, and
  reviewer impact.
- Authority-ledger routing that prevents old specs, chat, or external plans
  from silently governing implementation.

## 4. Non-Negotiable Constraints

- Alexandria owns the orchestration, custody, event, artefact, decision,
  reconciliation, review re-entry, and synthesis layers.
- Claude Agent SDK box harnesses are allowed and preferred for Grep-parity
  branch execution when they provide the needed filesystem, skill, MCP,
  sub-agent, context-isolation, and fork/resume behavior.
- Raw Anthropic Messages API calls remain available behind the provider-agnostic
  model client for bounded roles that do not require a Claude Agent SDK box.
- Alexandria does not use LangGraph as its core state-machine runtime.
- Any external agent runner transcript is an input to Alexandria's event and
  content-addressed custody layer. It is not the sole state of record.
- The model client remains provider-agnostic so Claude and GPT can serve
  different cognitive roles.
- The filesystem is working memory during research.
- The event log is append-only and complete for orchestrator I/O hops.
- Artefacts are immutable and content-addressed.
- Runtime/provider hidden state is never required to reconstruct a completed
  session.
- The first commercial wedge is pharma competitive intelligence.
- The system is not a database product.
- The system is not a general developer framework.
- Cross-session recursive improvement is preserved optionality, not a v1 product
  promise.
- Implementation work begins only after the relevant interface, schema, file
  format, control-flow, and proof obligations are specified.

## 5. Method Stack

This program is driven by four complementary methods.

### 5.1 Teleology-Preserving Planning

Mandatory loop:

1. Synthesize the intended system.
2. Inspect whether the synthesis misses the point.
3. Classify the miss.
4. Repair the governing surface.
5. Route to the smallest downstream planning, audit, evidence, interface, or
   bead-compilation protocol.

No complex plan, charter, spec, or bead queue lands while the governing point is
unstable.

### 5.2 Recursive Research Engine Design

Mandatory loop:

1. Define adequacy criteria at planning time.
2. Gather context before freezing the plan.
3. Spawn bounded research branches where parallel investigation is real.
4. Return pointer / analysis / evidence triplets.
5. Reconcile evidence, contradictions, and uncertainty.
6. Re-enter research when reviewers or thin evidence require it.
7. Synthesize through a single writer.

No report polish substitutes for research recursion.

### 5.3 Custody And Replay Proof Discipline

Mandatory loop:

1. Emit event for every orchestrator I/O hop.
2. Capture control decisions through tool calls.
3. Persist immutable content-addressed artefacts.
4. Link claims to admitted sources and intermediate work.
5. Validate replay from event log and artefact store.
6. Publish proof packs for claims that the system is allowed to make.

No major claim lands without reconstructible evidence.

### 5.4 Domain-Shaped Evidence Discipline

Mandatory loop:

1. Name the pharma CI question shape.
2. Define domain-specific source admissibility.
3. Define competitor, asset, trial, regulatory, commercial, and timeline axes
   where relevant.
4. Detect thin, stale, conflicting, and missing evidence.
5. Track unresolveds explicitly.
6. Render final report claims with custody.

No pharma CI claim is accepted because generic research prose sounds plausible.

## 6. Research Doctrine: Recursive Deep-Research Coordination

### 6.1 Problem Statement

High-stakes competitive intelligence fails when a research agent compresses too
early, loses source custody, smooths contradictions, or treats reviewer
criticism as editing feedback rather than research pressure.

Alexandria assumes that useful deep research needs working memory, branch
control, evidence admission, contradiction handling, synthesis discipline, and
re-entry.

### 6.2 Design Goal

Produce research reports whose claims can be traced, challenged, replayed, and
improved through recursive evidence work.

### 6.3 Failure Model

Primary failure classes:

- the planner routes before researching
- the task graph hides sequential dependencies
- sub-agents return prose without inspectable evidence
- the orchestrator reads everything and loses context economy
- contradiction handling becomes vague caveat prose
- reviewer findings do not alter the research state
- the writer stitches sections from incompatible voices
- event logs record activity while missing causal decisions
- claim custody exists only in the final report
- pharma domain adequacy remains generic

### 6.4 Planner-As-Researcher

The planner performs tool-mediated context gathering before emitting the Plan
File. The Plan File includes:

- research objective
- domain question shape
- adequacy criteria
- source-admission policy
- task graph
- dependency graph
- evidence-thin risks
- contradiction watchlist
- reviewer criteria

### 6.5 Recursive Orchestrator Loop

The orchestrator owns:

- inner-loop research task selection
- spawn / terminate / re-plan / deepen / ship decisions
- pointer-first reading of sub-agent returns
- adequacy assessment against the Plan File
- review-driven re-entry
- final session closure

Every control-flow decision is a decision-tool call with rationale and event
linkage.

### 6.6 Markdown Working Memory Economy

Markdown artefacts are the research workbench. They carry the durable state that
models cannot keep in context:

- plan files
- task graph files
- pointer files
- analysis files
- evidence files
- contradiction files
- review files
- TOC files
- section files
- report files

The orchestrator reads pointers first and pulls deeper files selectively.

### 6.7 Synthesis And Reconciliation

Synthesis is a first-class research act. It integrates admitted evidence,
contradictions, unresolveds, reviewer findings, and adequacy criteria into a
single research state.

Contradictions are represented as objects with:

- issue
- source set
- position set
- confidence state
- adjudication status
- unresolved status
- report treatment

### 6.8 Single-Voice Writer

One writer owns report voice and cohesion from table of contents through thesis
scaffold and section expansion. Research branches may supply evidence and
analysis. They do not own final section voice independently.

### 6.9 Reviewer-Driven Outer Loop

Reviewers and fact-checkers write reviews to files. The orchestrator reads the
reviews, classifies findings, and decides whether to ship or return to research
against planning-time adequacy criteria.
